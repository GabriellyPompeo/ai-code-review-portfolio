## 🤖 AI-Generated Code Under Review

The following TypeScript function was generated by an AI assistant when prompted:
> *"Write a generic TypeScript function that fetches data from an API and returns a typed response."*

```typescript
// AI-generated code
function fetchData(url: string): any {
  return fetch(url)
    .then(response => response.json())
    .then(data => data);
}

interface User {
  id: number;
  name: string;
  email: string;
}

const getUser = (id: number) => {
  return fetchData(`https://api.example.com/users/${id}`);
}

const processUsers = (users: any[]) => {
  return users.map(u => {
    return {
      id: u.id,
      fullName: u.name,
      contact: u.email
    };
  });
}
```

---

## ❌ Bugs Identified

### Bug #1 — Missing Generics (Type Safety Not Implemented) 🔴 CRITICAL

**Location:** `fetchData` function signature
**Problem:** The function is typed as returning `any`, completely defeating the purpose of TypeScript's type system. The AI was explicitly asked to write a **generic** function, but ignored the instruction.

```typescript
// ❌ AI generated this:
function fetchData(url: string): any { ... }

// ✅ Should be:
function fetchData<T>(url: string): Promise<T> { ... }
```

**Impact:** No type safety at call sites — errors only appear at runtime, not compile time.
**Category:** Instruction-Following Gap + Type Safety

---

### Bug #2 — Missing `async/await` (Wrong Return Type) 🔴 CRITICAL

**Location:** `fetchData` function body
**Problem:** The function returns a raw Promise chain but is not typed as `Promise<T>`. TypeScript will infer `Promise<any>` which propagates type unsafety throughout the codebase.

```typescript
// ❌ AI generated this (returns Promise<any> implicitly):
function fetchData(url: string): any {
  return fetch(url).then(response => response.json()).then(data => data);
}

// ✅ Should be:
async function fetchData<T>(url: string): Promise<T> {
  const response = await fetch(url);
  if (!response.ok) {
    throw new Error(`HTTP error: ${response.status}`);
  }
  return response.json() as Promise<T>;
}
```

**Impact:** No HTTP error handling — a 404 or 500 response will silently return an error object as data.
**Category:** Missing Edge Case + Silent Failure

---

### Bug #3 — `getUser` Not Using the Generic Type 🟠 HIGH

**Location:** `getUser` function
**Problem:** Even though a `User` interface exists, the `getUser` function doesn't use it. The return type is implicitly `any`, so TypeScript can't warn about property mismatches.

```typescript
// ❌ Current:
const getUser = (id: number) => {
  return fetchData(`https://api.example.com/users/${id}`);
}

// ✅ Should be:
const getUser = (id: number): Promise<User> => {
  return fetchData<User>(`https://api.example.com/users/${id}`);
}
```

**Impact:** TypeScript loses the ability to validate User property access throughout the codebase.
**Category:** Type Safety — Missed Opportunity

---

### Bug #4 — `processUsers` Uses `any[]` Instead of `User[]` 🟡 MEDIUM

**Location:** `processUsers` function parameter
**Problem:** The parameter `users: any[]` bypasses all type checking. If the API response changes shape, TypeScript won't catch it at compile time.

```typescript
// ❌ Current:
const processUsers = (users: any[]) => { ... }

// ✅ Should be:
const processUsers = (users: User[]) => { ... }
```

**Impact:** Refactoring risk — property renames won't be caught by the compiler.
**Category:** Type Safety

---

### Bug #5 — No Input Validation for `id` Parameter 🟡 MEDIUM

**Location:** `getUser` function
**Problem:** The `id` parameter accepts any number, including negative values and 0, which would result in invalid API calls.

```typescript
// ❌ Current:
const getUser = (id: number): Promise<User> => {
  return fetchData<User>(`https://api.example.com/users/${id}`);
}

// ✅ Should be:
const getUser = (id: number): Promise<User> => {
  if (!Number.isInteger(id) || id <= 0) {
    return Promise.reject(new Error('User ID must be a positive integer'));
  }
  return fetchData<User>(`https://api.example.com/users/${id}`);
}
```

**Impact:** Invalid API requests, potential server errors, poor user experience.
**Category:** Missing Edge Case

---

## ✅ Corrected Version

```typescript
// ✅ Fully corrected and type-safe version

async function fetchData<T>(url: string): Promise<T> {
  const response = await fetch(url);
  if (!response.ok) {
    throw new Error(`HTTP error ${response.status}: ${response.statusText}`);
  }
  return response.json() as Promise<T>;
}

interface User {
  id: number;
  name: string;
  email: string;
}

const getUser = async (id: number): Promise<User> => {
  if (!Number.isInteger(id) || id <= 0) {
    throw new Error('User ID must be a positive integer');
  }
  return fetchData<User>(`https://api.example.com/users/${id}`);
};

interface ProcessedUser {
  id: number;
  fullName: string;
  contact: string;
}

const processUsers = (users: User[]): ProcessedUser[] => {
  return users.map((u): ProcessedUser => ({
    id: u.id,
    fullName: u.name,
    contact: u.email,
  }));
};
```

## 💡 Key Takeaway

This is a common AI failure pattern: when asked for a **generic** TypeScript function, the AI generates a syntactically valid function that **avoids** generics by using `any`. This is an instruction-following gap combined with a type safety failure — the two most critical dimensions for TypeScript code evaluation.
